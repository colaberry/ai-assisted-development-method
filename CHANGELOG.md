# Changelog

All notable changes to this project are documented in this file.

The format follows [Keep a Changelog](https://keepachangelog.com/en/1.1.0/). The method document carries its own version scheme (v3.x.y); tooling and skills are versioned independently and listed in each release section.

## [Unreleased]

### Added

- **`sprint_gate.py` enforces declared scope via TASKS.md `Files:` allowlist** ([#27](https://github.com/colaberry/ai-assisted-development-method/issues/27)) — Method rule 11 ("silent scope expansion is an anti-pattern") becomes structural. Before this change, [tooling/hooks/sprint_gate.py](tooling/hooks/sprint_gate.py) only blocked writes under `sprints/vK/` when an earlier sprint was unlocked; a team could ship an entire vN+1 feature by writing to `src/`, `tests/`, and `migrations/` without touching `sprints/vN+1/`, and the hook would fire zero times. The hook now parses the active (unlocked, highest-numbered) sprint's `TASKS.md`, builds the union of `Files:` entries across open `[ ]` tasks — excluding `[x]` completed tasks and `[DEFERRED]` tasks — and when a prior sprint lacks `.lock` (the skip condition), blocks writes to files outside that allowlist. Directory entries (`src/auth/`) match descendants. Blocks are appended to `sprints/vN/.gate-blocks.log` so retros can review false positives and adjust task `Files:` declarations. Defensive defaults preserved: missing/unparseable `TASKS.md` warns and allows, and v1 bootstrap (no prior sprint) is unaffected. Tests in [tooling/tests/test_sprint_gate.py](tooling/tests/test_sprint_gate.py) (48/48 pass, +21 new). Closes a gap the 2026-04-20 multi-expert audit flagged between the README/diagram enforcement promise and the hook's real behavior.
- **`/security-review` and `/ui-qa` skills with `sprint_close.py` structural refusal** ([#26](https://github.com/colaberry/ai-assisted-development-method/issues/26)) — lands the "path (a)" fix for Method rule 6. CLAUDE.md rule 6 had named `/security-review` and `/ui-qa` as steps `/sprint-close` runs, but no skill files existed and `sprint_close.py` performed no corresponding artifact check — a sprint could lock cleanly while those steps were never run. The fix is the rule becoming real, not the rule being downgraded. Two new skills ship: [tooling/.claude/skills/security-review.md](tooling/.claude/skills/security-review.md) and [tooling/.claude/skills/ui-qa.md](tooling/.claude/skills/ui-qa.md), each producing a sprint-scoped artifact (`SECURITY-REVIEW.md` / `UI-QA.md`) from [tooling/templates/security-review-TEMPLATE.md](tooling/templates/security-review-TEMPLATE.md) and [tooling/templates/ui-qa-TEMPLATE.md](tooling/templates/ui-qa-TEMPLATE.md). [tooling/scripts/sprint_close.py](tooling/scripts/sprint_close.py) adds `check_scope_artifact`: parses the PRD's `` `/security-review` required: Yes|No `` (and equivalent `/ui-qa`) line, and when `Yes`, refuses to lock if the artifact is missing, missing its `Reviewer:` / `Date:` / `Decision:` fields, or carries `Decision: blocked`. `Decision: passed` and `Decision: n/a` are accepted passes. The unfilled template literal `Yes / No` is treated as "unspecified" (skip), so minimum-viable-adoption projects that use the PRD template verbatim are not broken. Tests in [tooling/tests/test_sprint_close.py](tooling/tests/test_sprint_close.py) cover parsing, per-check behavior, and end-to-end run_close scenarios (51/51 pass, +15 new). This turns Method rule 6 from aspirational to structural — the audit finding the rule cited in [#26](https://github.com/colaberry/ai-assisted-development-method/issues/26) is now resolved.
- **Semgrep security merge gate** ([#7](https://github.com/colaberry/ai-assisted-development-method/issues/7)) — [tooling/.github/workflows/security.yml](tooling/.github/workflows/security.yml) runs Semgrep (`--config=auto --severity=ERROR`) on every PR and push to main as a structural merge gate alongside [reconcile.yml](tooling/.github/workflows/reconcile.yml). Traceability + security are now the two structural gates; manual `/security-review` becomes the escalation path, not the only layer. Deliberate suppressions live in `docs/security/suppressions.md` (see [tooling/templates/security-suppressions-TEMPLATE.md](tooling/templates/security-suppressions-TEMPLATE.md)) with a 90-day re-review ceremony enforced by `state-check.py`. Handbook §1.9 "Security as a merge gate" documents the discipline.
- **`sprint_close.py` — atomic sprint closure** — [tooling/scripts/sprint_close.py](tooling/scripts/sprint_close.py) promotes the manual `/sprint-close` checklist to a script. Runs `reconcile.py --ci` (with optional `--strict-symbols` pass-through), verifies that `RETRO.md` is no longer the template stub (no `<placeholder>`, no literal `vN`, no literal `YYYY-MM-DD`, sections "What went well" and "What went poorly" contain real content), and verifies a sign-off (either an existing `SIGNOFF.md` with `Reviewer:` + `Date:` lines, or `--reviewer NAME` to create one). Writes `sprints/vN/.lock` only when every check passes — no partial closures. The `.lock` content records `locked_at`, `reviewer`, and `reconcile_status`, ready for downstream tooling (the planned PreToolUse hook will read this). Tests in [tooling/tests/test_sprint_close.py](tooling/tests/test_sprint_close.py) (23/23 pass).
- **Symbol-presence check in `reconcile.py`** — [tooling/scripts/reconcile.py](tooling/scripts/reconcile.py) now extracts candidate symbols from each task's title and `Acceptance:` line (backticked tokens are the strongest signal; bare `snake_case`/`camelCase`/`ALL_CAPS` identifiers are also picked up) and greps each task's `Files:` for them. A completed task whose listed files exist but contain none of those symbols gets a `STUB-WARNING:` note and is demoted to MEDIUM confidence — the "empty stub passes reconcile" pattern. New `--strict-symbols` flag elevates that pattern to a missing requirement (CI exit 1); off by default so existing pipelines don't break. Tests in [tooling/tests/test_reconcile.py](tooling/tests/test_reconcile.py) (21/21 pass). Method §/reconcile already promised this behavior — this lands the implementation.
- **`Autonomy:` annotation in TASKS.md** ([#8](https://github.com/colaberry/ai-assisted-development-method/issues/8)) — optional per-task subline (`direct` | `checkpoint` | `review-only`) that tunes how often `/dev` pauses for human confirmation. The level is chosen against task risk and existing test coverage, not how much you trust the engineer or the model: a doc fix is `direct`, an ordinary feature is `checkpoint` (the default), a payment-path or schema migration is `review-only`. [tooling/scripts/reconcile.py](tooling/scripts/reconcile.py) parses the annotation onto the `Task` dataclass (`autonomy`, `autonomy_valid` fields), validates against the three known values, and emits a stderr `WARNING` when the value is unrecognized — without failing the gate, since the annotation is advisory. [tooling/templates/sprint-TASKS-TEMPLATE.md](tooling/templates/sprint-TASKS-TEMPLATE.md) updated with examples of all three levels and the picking-the-level guidance. Handbook §1.10 "Tuning autonomy with `Autonomy:` annotations" covers when each level is appropriate and explicitly notes the annotation does not replace test coverage, the security gate, or the test matrix. Tests in [tooling/tests/test_reconcile.py](tooling/tests/test_reconcile.py) (29/29 pass, +8 new). The downstream `/dev` skill (when scripted) reads this field; until then the annotation is documentation for human reviewers.
- **`/incident` skill — post-deployment learning loop** ([#6](https://github.com/colaberry/ai-assisted-development-method/issues/6)) — [tooling/.claude/skills/incident.md](tooling/.claude/skills/incident.md) plus [tooling/templates/incident-TEMPLATE.md](tooling/templates/incident-TEMPLATE.md). Completes the SDLC coverage that today stops at sprint close. The skill orchestrates the post-mortem, pushes back on vague answers ("around 3pm", "some users", "communication was bad"), extracts a specific actionable prevention rule (rejects "write better tests" — demands something a test or linter could enforce), creates the cross-linked [docs/failures/](tooling/templates/failures-log-TEMPLATE.md) entry, and — critically — names the **enforcement surface** where the rule will actually run (CLAUDE.md / CI / architecture-guard test / security prompt / test-matrix discipline). A rule that lives only in the failures log will not catch the next occurrence. When the incident invalidates a requirement or ADR, the skill drafts a separate design-doc update PR or a superseding ADR; it never silently edits the spec. [state-check.py](state-check/scripts/state-check.py) adds `check_open_incidents()` which scans `docs/incidents/` for post-mortems missing a `Resolved:` timestamp and surfaces them as P1 learning-loop flags, with template-placeholder awareness (the template's `<YYYY-MM-DD HH:MM timezone>` doesn't count as resolved). Handbook §1.10 "Closing the loop after a production incident" covers the discipline and the three-strike rule. Tests in [state-check/tests/test_state_check.py](state-check/tests/test_state_check.py) (36/36 pass, +14 new).
- **PreToolUse hook — `sprint_gate.py`** — [tooling/hooks/sprint_gate.py](tooling/hooks/sprint_gate.py) is the structural complement to `sprint_close.py`. On every `Write`/`Edit`/`MultiEdit`/`NotebookEdit`, if the target file is under `sprints/vK/` and any earlier `sprints/vJ/` (J < K) is missing a `.lock` file, the hook blocks the operation with exit code 2 and a stderr message naming the unlocked sprints. This turns the anti-skip rule from cultural to structural: you cannot start writing into v2 while v1 is still open, no matter how confident you are that "v1 is basically done." Defensive by design — parse errors and unexpected conditions allow the operation (it's a reminder, not a security boundary). Install via [tooling/templates/claude-settings-hooks-TEMPLATE.json](tooling/templates/claude-settings-hooks-TEMPLATE.json) merged into the client repo's `.claude/settings.json`. Tests in [tooling/tests/test_sprint_gate.py](tooling/tests/test_sprint_gate.py) (27/27 pass). Closes the last structural-gate roadmap item — `/reconcile`, `sprint_close.py`, `security.yml`, and `sprint_gate.py` are now the four structural enforcement points.
- **Method document text fixes** — three method-level cleanups in [internal-mode/Internal_Product_Mode.md](internal-mode/Internal_Product_Mode.md) plus a new minimum-viable adoption path. (1) Stage 1 test-matrix contradiction resolved: Category D is now explicitly required wherever code touches existing production systems and optional otherwise — the table, the prose, and the anti-pattern now agree. (2) Gate 2 → 3 retention criterion tightened from "users keep coming back without being prompted" to a concrete pre-committed metric (default heuristic: ≥40% of Stage 2 users return self-directed in ≥3 of the 4 weeks following first meaningful use), with explicit guidance to pick metric and threshold before reading the data. (3) New [MINIMUM-VIABLE-ADOPTION.md](MINIMUM-VIABLE-ADOPTION.md) at repo root describes a four-piece adoption path (CLAUDE.md + stable IDs + `Satisfies:` + `reconcile.py` in CI) under a day for teams who can't take the full-bundle bootstrap; linked from [README.md](README.md) and [START-HERE.md](START-HERE.md).
- **Metrics Phase 1 — gate-event logger** ([#12](https://github.com/colaberry/ai-assisted-development-method/issues/12)) — new [`metrics/`](metrics/) module with [`metrics.py log-gate`](metrics/scripts/metrics.py) and `list-events` commands. The two structural-gate workflows ([reconcile.yml](tooling/.github/workflows/reconcile.yml), [security.yml](tooling/.github/workflows/security.yml)) now call `log-gate` as a final step with `if: always()`, so pass/fail records accumulate as JSONL in `docs/metrics/events.jsonl` without requiring engineer discipline. Phase 1 persistence has a known limitation (PR-branch events are lost on squash merge); durable commit-back lands in [Phase 2 (#13)](https://github.com/colaberry/ai-assisted-development-method/issues/13). Session-level token tracking, rework-reason logging, and retro auto-sections are deferred to Phase 2 — the threshold ranges that would interpret session data have not been validated against real teams.
- **AI code-smell PR review checklist** ([#5](https://github.com/colaberry/ai-assisted-development-method/issues/5), shipped via [PR #9](https://github.com/colaberry/ai-assisted-development-method/pull/9)) — pure-doc change targeting failure modes that pattern-match as "AI code smell": invented APIs, plausible-but-wrong docstrings, defensive coding for impossible states, silent scope creep beyond the task's `Satisfies:` IDs, tests modified to match implementation. [tooling/templates/code-review-AI-CHECKLIST.md](tooling/templates/code-review-AI-CHECKLIST.md) is the artifact reviewers attach to AI-assisted PRs — ~25 items across scope / correctness / error handling / comments / security / tests / fit / commit hygiene, with `[blocking]` items as merge-blockers. Handbook §1.8 "Reviewing AI-assisted PRs" explains why generic PR review misses these patterns and when to skip the checklist honestly (`AI-assisted: no` for tab-completion-only diffs). [.github/pull_request_template.md](.github/pull_request_template.md) adds the `AI-assisted: yes / no` field that points reviewers at the checklist. The fourth acceptance criterion from #5 — "at least 3 items cite a specific failure from this project's failures log" — is intentionally deferred until real engagement failures accumulate; until then the items are grounded in cross-project pattern observation rather than this repo's own history.
- **Metrics Phase 2 — partial ship** ([#13](https://github.com/colaberry/ai-assisted-development-method/issues/13)) — structural pieces only; threshold calibration deliberately deferred to a follow-up until real engagement data exists. [metrics/scripts/metrics.py](metrics/scripts/metrics.py) gains three commands: `log-session` (records a sprint-tagged work session with `--kind dev|tests|review|other`, optional `--task` ID, optional `--rework` flag), `count-sessions` (filterable by sprint, JSON-capable), and `list-events --event-type session`. [tooling/scripts/sprint_close.py](tooling/scripts/sprint_close.py) adds a new `check_sessions_logged` check: if the `metrics/` module is installed in the target repo and zero session events exist for the sprint being closed, `sprint_close.py` refuses to write `.lock`. Repos on the minimum-viable adoption path (no `metrics/`) are unaffected — the check passes with a "not installed" note. This turns session-logging discipline from cultural ("remember to log") to structural ("the sprint doesn't close without it"). [tooling/templates/retro-TEMPLATE.md](tooling/templates/retro-TEMPLATE.md) adds a Session metrics section populated from raw `list-events` output with an explicit "interpretation ranges not calibrated yet" warning. [tooling/.claude/skills/sprint-close.md](tooling/.claude/skills/sprint-close.md) documents the new refusal mode. [metrics/docs/METRICS.md](metrics/docs/METRICS.md) gains the `session` event schema with anti-pattern notes (no per-engineer field, no token auto-calc, no self-reported duration). **Not shipped:** threshold-based interpretation (healthy / high / low labels, rework-rate alerts, rollup CLI), automated retro-section generation, PR-commit-back for durable CI event persistence — all land in a follow-up once at least one real engagement has produced 3+ sprints of combined gate + session data. Tests: [metrics/tests/test_metrics.py](metrics/tests/test_metrics.py) 32/32 pass (+11), [tooling/tests/test_sprint_close.py](tooling/tests/test_sprint_close.py) 27/27 pass (+4).
- **Method skills bundle** — five conversational Claude Code skills wrapping the enforcement scripts, each gating on `state-check.py --json` preconditions before proceeding. [/prd](tooling/.claude/skills/prd.md) scopes a single sprint from the design document, refusing to start until the prior sprint is locked, and produces `sprints/vN/PRD.md` + `TASKS.md` whose IDs match the design doc and feed `reconcile.py`. [/dev](tooling/.claude/skills/dev.md) executes one task per session — non-negotiable — enforces the test-matrix categories named in `Tests required:`, reads the task's `Autonomy:` annotation to tune checkpoint cadence (`direct` end-to-end, `checkpoint` at milestones, `review-only` step-by-step), and refuses to mark a task `[x]` while symbol-presence warnings exist on it. [/sprint-close](tooling/.claude/skills/sprint-close.md) wraps `sprint_close.py`, walks the engineer through `RETRO.md` (rejecting template-stub answers structurally), orchestrates the security/UI-QA escalation when the PRD said they were required, and only then invokes the script that writes the `.lock` file. [/gate-1-to-2](tooling/.claude/skills/gate-1-to-2.md) and [/gate-2-to-3](tooling/.claude/skills/gate-2-to-3.md) run the Internal Product Mode graduation gates as evidence interviews — Stage 1→2 demands real-user usage outside the build team and a named 4–8 week commitment; Stage 2→3 enforces the structural pre-committed-retention rule (the metric and threshold must be quoted from the Stage 2 PRD, written *before* the data was read), names the engineering readiness initiative, the first commercial client, the support model, and full v3.x discipline. Each gate skill produces a written decision document at `docs/gates/`. Skills live next to [/incident](tooling/.claude/skills/incident.md) under [tooling/.claude/skills/](tooling/.claude/skills/).

### Planned

Prioritized roadmap — items listed in rough order of leverage-per-effort.

- **Metrics threshold calibration** (follow-up to [#13](https://github.com/colaberry/ai-assisted-development-method/issues/13)) — the structural pieces of Phase 2 shipped above (session logging, `sprint_close.py` refusal on zero sessions, retro template section). What remains is the *interpretation* layer: threshold ranges for healthy / high / low session counts, rework-rate alerts, tokens-per-session bands, automated retro-section generation from a `metrics-rollup.py retro-section` command, and PR-commit-back for durable CI event persistence across squash merges. **Structurally gated** on at least one real engagement completing ≥3 sprints under both Phase 1 gate logging and Phase 2 session logging — calibrating thresholds against simulated data produces metrics that mislead more than they inform. The deferral is per the issue's own precondition, not a scheduling preference.

## [3.2.1] — 2026-04-16

### Added

- **Phase 0** — client discovery and design-doc authoring before sprint work begins. Turns vague client input into signed-off requirements with assigned stable IDs. See [method/AI_Assisted_Development_Method_v3_2_1.md](method/AI_Assisted_Development_Method_v3_2_1.md).
- **Client intake template** ([tooling/templates/client-intake-TEMPLATE.md](tooling/templates/client-intake-TEMPLATE.md)) — 14 sections covering engagement metadata, problem framing, users and stakeholders, integrations, non-functional requirements, constraints, open questions, team context, risk assessment, commercial terms, and handoff readiness. `[REQUIRED]` fields block Phase 0 closure.
- **Internal Product Mode v1.0** ([internal-mode/Internal_Product_Mode.md](internal-mode/Internal_Product_Mode.md)) — companion process for internal product development that may eventually ship as SaaS. Three evidence-based stages (Exploration → Validation → Commercialization) with explicit graduation gates requiring written decision documents.
- **state-check v0.1** — paired CLI ([state-check/scripts/state-check.py](state-check/scripts/state-check.py)) and Claude Code skill ([state-check/.claude/skills/state-check.md](state-check/.claude/skills/state-check.md)) that detect repo mode, stage, active sprint, and flag P0/P1/P2 issues. Filesystem-derived detection (no separate config); heads-up display, not autopilot. First skill in the bundle.

### Notes

- v3.2.1 is a drop-in replacement for v3.2. Teams already on v3.2 can adopt Phase 0 incrementally on the next new engagement.
- Internal Product Mode is v1.0 intentionally, not v3.x — it is synthesized from general principles and has not yet been validated on a full engagement cycle. Treat as a starting point, not settled doctrine.
